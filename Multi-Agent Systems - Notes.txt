You probably have a good understanding of how to solve problems in uncertain situations in your job, and multi-agent systems allow us to structure these abstract processes so you can mainly focus on tweaking and editing natural language prompts instead of worrying about figuring out exact formulas or algorithms or dealing with messy data or confusing and complicated connections to disparate systems.

A multi-agent system is a computational program which can receive complicated natural language requests from users and respond agentically; that is, there is no strictly defined or pre-determined sequence of checks that an agent will go through in responding to a user question. In a multi-agent system, by defining agents with distinct goals and personalities, we can begin to approximate the kind of interactivity that occurs between humans working on a problem.

Let's start with an example everyone can relate to. Think about how a restaurant operates. You have hosts who greet customers and manage seating, waiters who take orders and deliver food, chefs who specialize in different types of cooking, busboys who clean up when customers leave, and managers who oversee the whole operation. Each person has a clear role, and they communicate with each other to deliver a complete experience. Multi-agent AI systems work in a similar way, and we can extend this thought process to essentially any kind of business system that operates based on well-defined roles and responsibilities.

Tools and Capabilities
To make multi-agent systems powerful, we need to make sure that the agents themselves can integrate with our business systems. This requires us to enable them with complex tools. We also need to learn how to route data and memories between agents as well as how to use agents to carefully augment requests to AI models with relevant context.

We're helping Uluru visitors get accurate cultural information in Arrernte and Pitjantjatjara languages. The system routes questions through language specialists who pull from a knowledge base and return translated, culturally appropriate responses.

How the Base System Works
Visitor asks a question through the input interface
Language Identification tool figures out what language they're using (acts like an orchestrator here)
Routes to either the Arrernte or Pitjantjatjara specialist
Specialist hits the Knowledge Base to grab relevant info
Response goes back through the chain, translated properly, to the visitor
It's a basic loop but gets the job done.

Extensions and Why They Matter
1. Translation Verification Agent

Sits between the specialists and final output to verify translations. Cultural nuances get lost easily - a word-for-word translation might be technically correct but miss the deeper meaning.

This agent catches anything that doesn't preserve cultural context before visitors see it. Both language specialists send their translations here first, then after verification it goes back to Language Identification for delivery.

2. Multimodal Media Tool

Pulls relevant images, audio clips, and videos to accompany text responses. Some cultural concepts are hard to explain with just words.

A photo of a sacred site, a recording of a traditional song, or video of a ceremony can communicate what paragraphs of text can't. Knowledge Base triggers it when relevant, and it feeds media back to both language specialists.

3. Cultural Sensitivity Checker

Filters knowledge to ensure we're not sharing sacred or restricted information. Some Indigenous knowledge is gender-specific, ceremonial, or just not meant for public sharing.

Without this guard rail, the system could accidentally violate cultural protocols. It sits between Knowledge Base and the specialists, acting as a gatekeeper.

4. Feedback Collector

Captures visitor ratings and comments about their experience. You can't improve what you don't measure.

This tells us which topics confuse people, what works well, and where we need better information. Directly linked to Visitor Input, runs independently so it doesn't slow down the main response flow.

Why It's Built This Way
Using an orchestrator pattern with Language Identification running the show. Each piece has one job (role specialization), and the orchestrator handles routing. Some steps have to be sequential (can't route before detecting language), but once that's done, tools like media and sensitivity checking can run in parallel.

Multi-Agent Architecture Implementation
In most frameworks, you'll encounter agents implemented as classes. You'll often inherit from a base Agent class and then fill in the specifics for that agent's unique logic, personality, and tools.

Agent Attributes and Prompts

The system prompt is a critical tool for enforcing specialization. A well-crafted prompt tells the agent:

Who you are: "You are a Data Validation Agent."
What you do: "Your sole purpose is to check incoming user data against predefined schemas."
What you don't do: "You do not perform calculations or interact with external APIs."
What tools you have: "You ONLY have access to the validate_schema tool."
Agent Tools

Tools give agents the ability to do things beyond just processing language, such as accessing databases, calling APIs, or performing calculations. Tool names and descriptions are critical metadata that the LLM uses to understand what the tool does and how to use it. It's also crucial to handle errors within your tools to prevent the agent from crashing.

Serialization and Deserialization

When agents communicate, they often exchange complex data objects. Since you can't directly send a Python object over a network, it needs to be serialized (converted to a string, usually JSON). The receiving agent then deserializes the string back into a usable object. Most modern frameworks handle this for you, but it's important to understand the process for debugging.


What is Smolagents framework?

Smolagents is a Python framework for building multi-agent systems powered by large language models (LLMs). It is designed to make it easy to create, orchestrate, and manage agents that can use tools (Python functions), maintain state, and collaborate to solve complex tasks.

Smolagents lets you build intelligent, tool-using, multi-agent systems with LLMs and Python, making it easy to combine language understanding with real-world actions and stateful workflows.

How does it work?
Define Tools: Decorate Python functions with @tool to make them available to agents.
Create Agents: Subclass ToolCallingAgent and specify which tools the agent can use.
Orchestrate: Use agents in your application, passing prompts and letting them call tools as needed.
LLM-Powered Reasoning: The agent framework parses the LLM's output, detects tool calls, and executes the corresponding Python functions.
What is a ToolCallingAgent?
ToolCallingAgent is a core class in the Smolagents framework. It represents an agent powered by a large language model (LLM) that can:

Interpret prompts and user input
Decide when to call Python functions ("tools")
Execute those tools and use their results in their reasoning
How does it work?
Define tools: Decorate Python functions with @tool.
Create an agent: Instantiate or subclass ToolCallingAgent, passing the tools and an LLM model.
Interact: Call .run(prompt) on the agent.
The agent will:
Use the LLM to interpret the prompt.
Decide if a tool should be called.
Call the tool if needed, and use its result in the response.
from smolagents import ToolCallingAgent, OpenAIServerModel, tool
import json

# Initialize a small language model
model = OpenAIServerModel(
    model_id="gpt-4o-mini",  # Small, efficient model
    api_key="your-api-key"
)

# Define tools that agents can use
@tool
def get_weather(city: str) -> str:
    """Get weather information for a city."""
    return f"Weather in {city}: Sunny, 75°F"

@tool
def save_data(key: str, value: str) -> str:
    """Save data to memory."""
    memory[key] = value
    return f"Saved: {key} = {value}"

# Global shared state
memory = {}

# Create a SmoLAgent
class WeatherAgent(ToolCallingAgent):
    def __init__(self):
        super().__init__(
            tools=[get_weather, save_data],  # Tools this agent can use
            model=model,                     # Small language model
            name="weather_assistant"         # Agent identifier
        )

    def ask_weather(self, city: str):
        """Agent processes a weather request."""
        response = self.run(
            f"Get weather for {city} and save it. "
            f"Respond with JSON: {{'weather': 'description', 'saved': true/false}}"
        )
        return response

# Usage
agent = WeatherAgent()
result = agent.ask_weather("New York")
print(result)  # Agent uses tools automatically to get and save weather
